# 系统整体架构

本文档基于当前仓库中的真实代码实现整理，不以历史方案稿为准。目标是说明系统当前如何运行、各层承担什么职责，以及当前架构的稳定边界与耦合点。

## 背景

项目当前是一个桌面实时同传应用，目标是同时支持两条业务链路：

- Channel 1：麦克风中文输入，经火山引擎实时翻译后输出英文语音。
- Channel 2：系统音频中的英文输入，经火山引擎实时翻译后输出中文字幕。

当前实现采用 Tauri 桌面壳 + React 前端 + Python sidecar + Python 翻译核心 的组合形态。桌面端负责窗口与进程生命周期，Python 负责实时音频与翻译业务。

## 总体分层

系统实际运行时可以分为四层：

1. 桌面宿主层  
由 `src-tauri/` 提供。负责拉起 sidecar、传递资源路径、管理主窗口与字幕浮窗生命周期。

2. 前端交互层  
由 `src/` 提供。负责配置页、日志页、字幕浮窗页面，以及三路 WebSocket 的客户端连接。

3. 后端控制层  
由 `desktop_backend/` 提供。负责配置管理、环境诊断、运行编排、日志广播、字幕广播和 WebSocket 命令分发。

4. 翻译引擎层  
由 `main.py`、`core/`、`gui/` 提供。负责音频采集、火山引擎协议交互、音频播放、字幕聚合与 CLI 模式兼容展示。

## 当前运行拓扑

```text
[用户]
  │
  │ 配置/启动/停止
  ↓
[React 主窗口]
  │
  │ invoke('launch_sidecar')
  ↓
[Tauri 宿主]
  │
  │ 拉起 Python sidecar
  ↓
[desktop_backend/sidecar.py]
  ├── /ws/control   ← 配置与运行控制
  ├── /ws/logs      ← 日志推送
  └── /ws/subtitle  ← 字幕推送
          │
          ↓
 [RuntimeService]
          │
          ↓
 [DualChannelTranslator]
   ├── Channel 1：麦克风采集 → 火山引擎 s2s → 音频输出
   └── Channel 2：系统音频采集 → 火山引擎 s2t → 字幕聚合
```

## 真实代码映射

当前实现与目录的主要映射关系如下：

- 桌面宿主：`src-tauri/src/lib.rs`、`src-tauri/tauri.conf.json`
- 前端交互：`src/App.tsx`、`src/hooks/useWebSocket.ts`、`src/pages/*`
- WebSocket 入口：`desktop_backend/sidecar.py`
- 配置/诊断/运行编排：`desktop_backend/services.py`
- 翻译主控：`main.py`
- 音频采集：`core/audio_capture.py`、`core/system_audio_capture.py`
- 火山引擎协议：`core/volcengine_client.py`
- 音频输出：`core/audio_output.py`
- 日志与观测：`core/logging_utils.py`
- CLI 字幕兼容：`gui/subtitle_window.py`

## 业务模块视角

从业务能力而不是代码目录来看，当前系统更稳定的拆分方式是以下八个模块：

1. 配置管理模块
2. 环境诊断模块
3. 运行编排模块
4. 音频采集模块
5. 火山引擎对接模块
6. 音频输出模块
7. 字幕处理与展示模块
8. 桌面宿主与交互模块

日志能力横跨所有模块，属于横切能力，不单独作为主业务模块。

## 关键架构决策

### 1. 桌面端采用 Tauri，翻译核心保留 Python

当前没有把实时音频和翻译链路迁移到 Rust 或前端，而是保留 Python 作为业务核心。这样做的好处是复用现有翻译实现、减少重写成本。代价是桌面宿主与业务核心之间必须通过 sidecar 进程边界通信。

### 2. 前后端通信采用三路 WebSocket

当前不是单一 RPC 通道，而是明确拆成三路：

- `/ws/control`：命令与响应
- `/ws/logs`：日志流
- `/ws/subtitle`：字幕流

这个设计使控制流和推送流分离，简化了前端处理逻辑，也降低了一个通道同时承担多类消息时的复杂度。

### 3. 双通道翻译业务仍由一个主控对象持有

`DualChannelTranslator` 目前统一持有 Channel 1 和 Channel 2 的流程，便于集中启动和停止。但这也让它同时承担了编排、字幕聚合、设备初始化等多种职责，是当前最明显的高耦合点。

## 当前边界

当前架构中已经比较清晰的边界有：

- 宿主层和业务核心之间通过 sidecar 进程边界隔离。
- WebSocket 入口与具体业务逻辑通过 service 调用隔离。
- 火山引擎协议细节主要收敛在 `VolcengineTranslator`。
- 字幕展示层已经支持 Tauri 浮窗和 Tkinter 两套实现。

## 当前耦合点

### 1. `DualChannelTranslator` 职责过重

它当前同时承担：

- 设备初始化
- 双通道主循环
- 字幕聚合
- Tkinter 兼容处理
- 统计信息输出

这意味着“翻译主控”“字幕处理”“运行编排”还没有完全解耦。

### 2. `services.py` 只是逻辑聚合，不是文件级解耦

虽然 `ConfigService`、`HealthService`、`RuntimeService` 已经形成逻辑边界，但它们仍然放在同一文件中。当前是“职责已分离、文件未分离”的状态。

### 3. 字幕处理逻辑仍放在主翻译器中

字幕聚合规则 `_append_ch2_fragment()` 和 `_flush_ch2_buffer()` 现在位于 `main.py` 中，这说明字幕处理还是运行主控的一部分，而不是独立模块。

## 风险与限制

- 当前系统对 Windows 音频环境依赖较强，尤其依赖立体声混音或 VB-CABLE。
- 构建形态中 Python bundle 的资源嵌入仍有历史问题，说明部署链路还未完全稳定。
- 前端目前主要消费后端提供的现成运行态，没有独立的领域状态层，业务规则仍偏后端中心。

## 后续演进建议

建议下一阶段按业务模块继续解耦，而不是按目录重命名：

1. 将字幕聚合逻辑从 `DualChannelTranslator` 中抽成独立服务。
2. 将 `services.py` 拆成独立服务文件，保持当前职责边界。
3. 将火山引擎协议层与运行编排层的异常处理策略分离。
4. 为业务模块补自动化测试，而不是只保留运行产物和缓存文件。
