# 运行链路架构说明

本文档描述系统当前的真实运行链路，重点说明启动、控制、翻译、日志、字幕和退出过程如何在各层之间流转。

## 1. 启动链路

启动链路从前端页面开始，由 Tauri 拉起 Python sidecar。

```text
[React App]
  │
  │ invoke('launch_sidecar')
  ↓
[src-tauri/src/lib.rs]
  │
  │ 选择运行模式
  ├─ 生产模式：查找 bundle/python/rt-engine.exe
  └─ 开发模式：回退到系统 python
  │
  │ 启动 desktop_backend/sidecar.py
  ↓
[sidecar.py]
  │
  │ 绑定 127.0.0.1:0
  │ stdout 输出 {"ready": true, "port": N}
  ↓
[Tauri 收到端口]
  │
  ├─ emit sidecar-ready
  └─ 显示 subtitle-overlay 窗口
  ↓
[前端 useWebSocket 连接三路 WS]
```

当前实现中，Tauri 还会在生产模式下注入 `PYTHONPATH` 和 `RT_APP_DATA`，用于让 sidecar 正确定位依赖和配置目录。

## 2. 控制链路

控制链路负责承接前端的配置和运行命令。

```text
[ConfigPage / StatusBar]
  │
  │ sendCommand(cmd, payload)
  ↓
[useWebSocket]
  │
  │ /ws/control
  ↓
[sidecar.handle_control()]
  │
  │ dispatch_command(cmd, payload)
  ↓
[ConfigService / DeviceService / HealthService / RuntimeService]
  │
  ↓
[返回 {id, ok, data|error}]
```

当前支持的主命令包括：

- `load_config`
- `save_config`
- `scan_devices`
- `test_connection`
- `env_check`
- `start`
- `stop`
- `status`

## 3. Channel 1 业务链路

Channel 1 负责“中文语音输入，英文音频输出”。

```text
[麦克风设备]
  ↓
[AudioCapturer]
  ↓
[DualChannelTranslator.channel1_loop]
  ↓
[VolcengineTranslator(mode=s2s)]
  ↓
[返回英文文本 + 英文语音]
  ↓
[OggOpusPlayer]
  ↓
[VB-CABLE / 输出设备]
```

这里的关键点是：

- 音频采样块由 `AudioCapturer.get_chunk()` 拉取。
- 火山引擎返回的音频数据由 `OggOpusPlayer` 解码并播放。
- 文本结果当前主要用于日志和统计，不是主展示出口。

## 4. Channel 2 业务链路

Channel 2 负责“英文音频输入，中文字幕输出”。

```text
[系统音频 / 立体声混音]
  ↓
[SystemAudioCapturer]
  ↓
[DualChannelTranslator.channel2_loop]
  ↓
[VolcengineTranslator(mode=s2t)]
  ↓
[英文或中文文本片段]
  ↓
[字幕聚合逻辑]
  ↓
  ├─ 桌面模式：subtitle_callback → /ws/subtitle → React 浮窗
  └─ CLI 模式：SubtitleWindowThread → Tkinter 字幕窗
```

这里的关键点是：

- Channel 2 不输出音频，只输出字幕。
- 字幕片段会先经过聚合、去重和 `flush` 规则。
- 桌面模式与 CLI 模式共用同一套字幕聚合输入，但展示端不同。

## 5. 日志链路

日志不是附属功能，而是当前系统的重要观测路径。

```text
[各业务模块]
  ↓
[ChannelLogger]
  ↓
[realtime_translator logger]
  ├─ FileHandler
  ├─ StreamHandler
  └─ WSLogHandler
         ↓
       /ws/logs
         ↓
      [LogPage]
```

`WSLogHandler` 会保留环形历史缓冲，因此新连接的日志页面可以先拿到历史日志，再接收增量推送。

## 6. 状态广播链路

运行状态既可以轮询，也会在状态变化时主动推送。

```text
[RuntimeService]
  │
  ├─ status 属性供轮询读取
  └─ on_state_change 回调
          ↓
    [sidecar 广播到 /ws/control]
          ↓
    [前端 handleStateChange]
          ↓
    [StatusBar / 页面状态]
```

当前前端仍保留每 3 秒一次的 `status` 轮询，因此状态同步属于“事件推送 + 兜底轮询”的混合方案。

## 7. 退出链路

退出过程由主窗口关闭触发，Tauri 主动回收 sidecar。

```text
[主窗口关闭]
  ↓
[Tauri on_window_event]
  ↓
[kill sidecar child]
  ↓
[进程退出]
```

此外 sidecar 内部还有 watchdog 线程：

- 如果检测到父进程消失，会主动 `os._exit(1)`。
- 这保证了宿主异常退出时，Python sidecar 不会长期残留。

## 8. 当前链路上的关键观察

### 1. 启动链路已比较清晰

桌面宿主、sidecar、前端连接三段职责分明，边界相对稳定。

### 2. 翻译链路的业务责任仍集中在 `DualChannelTranslator`

当前 Channel 1 和 Channel 2 虽然业务目标不同，但仍由同一个主控对象直接管理，因此运行链路稳定，模块边界仍偏粗。

### 3. 字幕链路已经具备独立演进基础

字幕输入、字幕聚合、字幕展示已经能从链路上区分，下一步可以把聚合逻辑继续抽离成单独业务模块。
