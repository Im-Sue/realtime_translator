# CH1 音频突发丢包与 rebuffer 风暴 — 复盘

> 日期: 2026-04-03
> 影响: Channel 1 英文语音严重卡顿，对方仅能听到 1-2 秒片段
> 根因: 环形缓冲区容量与 API 推送模式不匹配 + rebuffer 阈值过于激进

---

## 现象

- 用户说中文，对方只能听到约 1-2 秒英文语音，随后静音
- 诊断日志显示：74 秒会话中收到 36.6s 音频，仅播放 13.4s，**丢弃 23.2 秒**
- 状态机在 PLAYING 和 REBUFFERING 之间频繁切换（10 次 rebuffer）

关键日志片段:

```
CH1播放诊断: state=REBUFFERING 缓冲=0.0ms 样本=0
  输入包=117 输入=3513284B 已写=36.60s 已播=13.40s
  underflow=20 rebuffer=10 dropped=2226884
```

---

## 原因

### 1. 缓冲区容量与 API 推送模式的根本矛盾

火山引擎 s2s API 的音频推送模式是**句级突发**：

- 每次翻译完一句话，集中推送该句的全部音频（通常 1-5 秒）
- 多个音频包间隔仅 0.1ms（近乎瞬时到达）
- 句与句之间有数秒静默期

而环形缓冲区的设计假设是**匀速流式到达**：

- 初始容量 = `48000 × 2ch × 0.5s = 48000 样本`（500ms）
- 溢出策略 = 丢弃最旧数据（`_ring_write` 中 `overflow > 0` 分支）

**矛盾**：1 秒的突发需要 96000 个 stereo 样本的空间，但缓冲区只有 48000。每次突发必然溢出，丢弃约一半数据。

数学验证（首次突发）:

```
收到: 86784 bytes = 43392 mono样本 → stereo上混 = 86784 样本 (0.9s)
缓冲: 48000 样本容量 (0.5s)
溢出: 86784 - 48000 = 38784 样本被丢弃 (0.4s)
实际保留率: 55%
```

随着音频累积，保留率持续下降:

| 时间点 | 收到 | 播放 | 丢弃 | 保留率 |
|--------|------|------|------|--------|
| +20s   | 0.9s | 0.56s | 0.34s | 62% |
| +25s   | 5.5s | 2.28s | 3.26s | 41% |
| +35s   | 10.5s | 4.08s | 6.40s | 39% |
| +60s   | 36.6s | 13.4s | 23.2s | 37% |

### 2. rebuffer 阈值过于激进

状态机转换规则:

```
PLAYING → REBUFFERING: 连续 2 次 underflow（即 40ms 无数据）
REBUFFERING → PLAYING: 缓冲达到 100ms (resume_watermark)
```

**问题**：句级推送模式下，句间间隔（2-5 秒静默）**必然**导致 underflow。40ms 的容忍度远不够——每次句间间隔都会触发 rebuffer，而 rebuffer 期间新到达的音频被静音输出浪费，进一步加剧丢包。

### 3. 恶性循环

两个问题互相放大:

```
突发到达 → 缓冲区溢出 → 丢掉大部分音频
  → 播放仅存的小段 → 很快播完 → underflow
  → 2次underflow → 进入REBUFFERING → 输出静音
  → 下次突发到达 → 又溢出 → 再丢掉大部分
  → 循环...
```

用户听到的效果：每句话只有开头 1-2 秒，然后静音到下一句。

---

## 解决

### 修复 1: 扩大缓冲区容量

```python
# 修改前: 500ms
self._capacity = max(1, int(self.output_rate * self.channels * 0.5))

# 修改后: 30秒 (可容纳连续多句突发)
self._capacity = max(1, int(self.output_rate * self.channels * 30))
```

- 30 秒 @ 48kHz stereo = 2,880,000 int16 样本 ≈ **5.5MB 内存**
- 即使连续 10 句话的音频同时到达（极端场景），也不会溢出
- 内存开销可忽略不计

### 修复 2: 放宽 rebuffer 触发阈值

```python
# 修改前: 连续 2 次 underflow (40ms) 就 rebuffer
if self._underflow_count >= 2:

# 修改后: 连续 10 次 underflow (200ms) 才 rebuffer
if self._underflow_count >= 10:
```

- 句间间隔 200ms 以内视为正常波动，不触发 rebuffer
- 真正的数据断流（> 200ms 连续无数据）才进入 rebuffer 保护
- 大幅减少无谓的状态切换和静音输出

## 预防

### 设计原则

1. **了解上游推送模式再设计缓冲区**

   不要假设"流式 = 匀速到达"。需要先确认上游 API 的实际推送模式：
   - 匀速流式（如 WebRTC）→ 小缓冲区 (100-500ms) 足够
   - 句级突发（如同传 TTS）→ 缓冲区必须能容纳完整句子 (5-30s)
   - 混合模式 → 按最差情况设计

2. **溢出策略要匹配业务语义**

   `_ring_write` 的 "丢弃最旧数据" 策略在匀速场景合理（丢旧数据保持实时性），但在突发场景灾难性——因为"最旧数据"恰恰是当前句子的开头，丢掉后用户听到的是半截话。

   **替代方案**：对于突发场景，应优先选择"阻塞/限速"或"无限队列 + 回压"策略。

3. **rebuffer 阈值要匹配数据到达间隔**

   rebuffer 的触发阈值必须 > 数据的正常到达间隔。如果句间间隔是 2-5 秒，那 40ms 的 underflow 容忍度完全不合理。

   经验值:
   - 匀速流式: 2-5 次 underflow (20-100ms)
   - 句级突发: 10-50 次 underflow (200ms-1s)

### 排查 checklist

遇到"音频只播放片段"问题时，优先检查:

- [ ] 日志中 `dropped` 样本数是否 > 0（缓冲区溢出）
- [ ] `rebuffer` 次数是否异常高（阈值过激进）
- [ ] `已写` vs `已播` 的差距是否持续增大（数据被丢弃/静音浪费）
- [ ] 上游音频包的 `间隔` 是突发 (< 1ms) 还是匀速 (10-100ms)
- [ ] 缓冲区容量 vs 单次突发数据量的比值（应 > 5x）

### 关键指标

健康状态下的诊断日志应满足:

```
dropped = 0              # 无溢出丢弃
rebuffer <= 1             # 极少 rebuffer
已写 ≈ 已播 (差距 < 2s)   # 数据被及时消费
```
