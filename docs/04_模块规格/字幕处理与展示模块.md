# 字幕处理与展示模块

## 模块定位

字幕处理与展示模块负责把 `CH2` 的实时识别 / 翻译结果转成可稳定消费的字幕事件，再按运行模式输出到桌面浮窗或 CLI 兼容窗口。

## 当前实现

当前由两层组成：

### 字幕处理层

位于 `main.py`，核心逻辑包括：

- `_start_ch2_sentence()`
- `_update_ch2_buffer()`
- `_emit_ch2_subtitle()`
- `_finish_ch2_sentence()`
- `_flush_stale_ch2_sentence()`
- `_handle_ch2_subtitle_result()`

### 字幕展示层

位于：

- `src/components/SubtitleOverlay.tsx`
- `src/components/subtitleFlow.ts`
- `gui/subtitle_window.py`

其中 React 浮窗是桌面端主路径，Tk 窗口是 CLI 兼容路径。

## 主要职责

- 根据火山事件识别一句字幕的开始、流式更新和结束
- 对英文和中文内容分别维护当前句缓冲
- 在桌面模式下广播统一字幕事件到 `/ws/subtitle`
- 在 CLI 模式下回退到 Tk 窗口显示
- 根据超时策略补发结束事件，避免最后一句卡住

## 当前事件模型

当前桌面字幕不再依赖旧的“片段拼接 + flush 文本协议”，而是统一输出：

- `start`
- `streaming`
- `end`

消息结构为：

```json
{
  "type": "streaming",
  "en": "How are you",
  "zh": "你好吗",
  "is_final": false
}
```

## 核心规则

### 句级状态机

`CH2` 当前句会维护：

- `ch2_en_buffer`
- `ch2_zh_buffer`
- `ch2_sentence_active`
- `ch2_source_completed`
- `ch2_translation_completed`

只有在合适的生命周期事件抵达时，才会真正结束当前句并推送最终字幕。

### 超时兜底

如果上游没有收到 `End` 事件，但在一段时间内也没有新的更新，`_flush_stale_ch2_sentence()` 会主动补发结束，避免浮窗一直停留在流式状态。

### 展示层再归约

桌面浮窗内部不会直接把每条事件原样渲染，而是交给 `subtitleFlow` 做二次归约，形成：

- 历史段落
- 当前句
- 正在流式追加的尾部文本

这样可以同时兼顾“持续可读”和“实时变化感”。

## 上下游边界

### 上游

- `VolcengineTranslator`
- `DualChannelTranslator`

### 下游

- `/ws/subtitle`
- `SubtitleOverlay`
- `gui.subtitle_window`

## 当前限制

- 字幕处理状态机仍嵌在 `DualChannelTranslator` 内，尚未独立成服务对象。
- 英中语言判定仍依赖字符分布启发式，不是严格语种检测。
- 当前模型只覆盖 `CH2` 字幕路径，`CH1` 结果主要仍以音频输出为主。
