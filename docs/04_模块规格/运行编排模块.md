# 运行编排模块

## 模块定位

运行编排模块负责把桌面端的“启动 / 停止 / 查询状态”请求转成翻译引擎生命周期管理，是 sidecar 内最靠近业务主控的一层服务。

## 当前实现

当前实现主要位于 `desktop_backend/services.py` 的 `RuntimeService`，并通过 `main.py` 中的 `DualChannelTranslator` 执行真实翻译链路。

## 主要职责

- 管理翻译器实例和后台 `asyncio.Task`
- 维护运行状态 `running / ch1 / ch2 / uptime`
- 在启动前校验火山引擎凭据
- 注入字幕回调和状态广播回调
- 负责翻译器的启动、停止和异常收尾

## 输入与输出

### 输入

- `/ws/control` 的 `start`
- `/ws/control` 的 `stop`
- `/ws/control` 的 `status`
- 配置管理模块返回的完整配置

### 输出

- 运行状态对象：`{ running, ch1, ch2, uptime }`
- 启停结果
- 状态变化广播事件

## 核心流程

```text
前端 start
  -> RuntimeService.start()
  -> 校验 app_key / access_key
  -> 将当前配置写入临时 YAML
  -> 构造 DualChannelTranslator(subtitle_callback=...)
  -> create_task(_run)
  -> 状态切换为 running
```

停止流程：

```text
前端 stop
  -> RuntimeService.stop()
  -> 取消后台 task
  -> 调用 translator.stop()
  -> 清空运行状态
  -> 广播 state_change
```

## 关键实现特点

### 临时 YAML 桥接

`DualChannelTranslator` 目前仍然以配置文件为主入口，因此运行编排层在启动时会把当前配置写入临时 YAML，再把文件路径传给主控对象。

这是一种“兼容现有主控”的桥接方案，优点是改动小，缺点是还没有形成真正的领域对象接口。

### 字幕回调注入

桌面模式下，`RuntimeService` 会把 `on_subtitle` 回调注入到 `DualChannelTranslator`，这样：

- 桌面浮窗可以直接消费 `/ws/subtitle`
- `main.py` 不需要初始化 Tkinter 字幕窗口

### 状态模型偏粗粒度

当前 `ch1` 和 `ch2` 状态在启动时都会直接置为 `running`，异常时同时置为 `error`。这更接近“整体运行态”，还不是精细化通道健康模型。

## 上下游边界

### 上游

- `desktop_backend/sidecar.py` 的命令分发

### 下游

- `DualChannelTranslator`
- 字幕广播回调
- 状态广播回调

## 当前限制

- 仍通过临时文件桥接配置，不够优雅。
- `ch1`、`ch2` 运行状态不是来自通道级真实健康检查。
- 编排层仍然依赖一个偏重的主控对象，无法独立控制单通道启停。
