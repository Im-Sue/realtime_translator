# 配置管理模块

## 模块定位

配置管理模块负责应用配置的默认值、加载、补齐、脱敏和持久化，是桌面前端与翻译引擎之间的静态配置中枢。

## 当前实现

当前实现位于 `desktop_backend/services.py`，核心对象与函数包括：

- `DEFAULT_CONFIG`
- `_deep_merge_config()`
- `_resolve_config_dir()`
- `ConfigService`

## 主要职责

- 提供桌面端首启可用的默认配置模板。
- 读取 `config.yaml`，并把旧配置补齐到最新字段结构。
- 保存前端提交的配置，并保留真实密钥避免被脱敏值覆盖。
- 向前端返回脱敏后的安全配置。
- 根据开发态/生产态决定配置文件实际存放路径。

## 输入与输出

### 输入

- `/ws/control` 的 `load_config`
- `/ws/control` 的 `save_config`
- Tauri 生产模式注入的 `RT_APP_DATA`
- 当前进程环境中的 `APPDATA`

### 输出

- 返回给前端的安全配置对象
- 持久化到磁盘的完整 `config.yaml`

## 当前配置结构重点

当前默认配置里，和本分支最相关的字段是：

```yaml
audio:
  vbcable_output:
    sample_rate: 48000
    target_format: pcm
    use_ffmpeg: true
```

这意味着：

- 桌面端默认以 `PCM 48kHz` 作为 `CH1` 输出主路径。
- 旧配置文件即使没有 `target_format` 字段，加载后也会被自动补齐。

## 核心规则

### 路径决策

优先级如下：

1. `RT_APP_DATA`
2. 打包模式下 `%APPDATA%/RealtimeTranslator`
3. 开发模式下项目根目录

### 深度补齐

加载和保存都会经过 `_deep_merge_config()`，目的不是“覆盖用户设置”，而是“补齐新增字段”，避免：

- 前端类型已经升级，但旧配置缺字段
- 后端读取新字段时出现默认值不一致
- 保存一次旧配置后又把新字段丢掉

### 密钥脱敏

- 返回前端时，`app_key` 和 `access_key` 会被替换为带 `****` 的脱敏值。
- 前端保存时如果仍然传回脱敏值，后端会保留原始密钥，不做覆盖。

## 上下游边界

### 上游

- `ConfigPage`
- `RuntimeService`
- `HealthService`

### 下游

- 本地文件系统
- 应用数据目录

## 当前限制

- 仍然使用单一 YAML 文件，没有配置版本号和显式迁移脚本。
- 目前没有独立的配置校验层，字段合法性主要依赖前端表单约束和运行时校验。
- 字段结构直接贴近 UI 和运行时对象，领域抽象还不够强。
